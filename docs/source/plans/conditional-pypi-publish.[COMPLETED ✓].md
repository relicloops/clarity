# Conditional PyPI Publish

## Goal

Today any `v*` tag push triggers a PyPI upload through
`.github/workflows/ci.yml`. Git tags are used by this project as
history milestones -- they are bumped for README edits, plan changes,
Makefile tweaks, CHANGELOG regenerations, audits, and so on. Only a
subset of those tags actually changed theme code that consumers of
the `sphinx-clarity` wheel would care about.

We need the CI to:

1. Still run on every `v*` tag (so the build stays green and the tag
   is recorded as a milestone).
2. Publish to PyPI **only** when code that ships in the wheel changed
   since the previous tag. A tag that only touches `docs/`, `audits/`,
   `plans/`, the Makefile, or the README must not create a new PyPI
   release.
3. Create a GitHub release only when the wheel is actually published
   (otherwise the release page and the wheel drift apart).
4. Support an explicit override for edge cases (e.g. republish an
   identical wheel because the previous publish failed on PyPI).

---

## What counts as a "publish-worthy" change

The wheel is built from `src/clarity/` and described by `pyproject.toml`.
Everything else lives outside the wheel. The publish-trigger paths are:

```
src/clarity/**
pyproject.toml
```

Any other path change -- `docs/**`, `README.md`, `CHANGELOG.md`,
`cliff.toml`, `.github/**`, `Makefile`, `docs/Makefile`, `.prompt/**`,
`plans/**`, `audit/**`, `LICENSE`, `.gitignore`, etc. -- does NOT
warrant a new wheel.

Rationale:

- `pyproject.toml` is in the list because the `version` field there is
  the PEP 440 string PyPI sees. A version-only change with zero code
  delta is technically a no-op for consumers, but it is the canonical
  "I want a new wheel" signal the maintainer is sending. Keep it in.
- `CHANGELOG.md` is regenerated by `git-cliff` and often bumped in the
  same commit as the version. Its content does not ship inside the
  wheel -- exclude.
- `LICENSE` is repo-root only; `wheel` packages are declared in
  `pyproject.toml` under `tool.hatch.build.targets.wheel.packages`
  and only include `src/clarity`. Exclude.

## Trigger logic (proposed)

Add a lightweight **gate job** at the start of `ci.yml` that decides
whether `publish` runs. The gate:

1. Checks out with `fetch-depth: 0` so tags are available.
2. Resolves the previous tag: `git describe --tags --abbrev=0 HEAD^`.
   - If there is no previous tag (first release), treat as "publish".
3. Diffs the current tag against the previous tag:
   `git diff --name-only <prev> <this>`.
4. Filters the diff against the publish-trigger path list.
5. Reads the tag annotation for overrides:
   - `[force-publish]` -> publish even if no paths matched.
   - `[skip-publish]`  -> never publish, even if paths matched.
6. Outputs a boolean step-output: `should_publish`.

Downstream jobs (`publish`, `github-release`) gate on
`needs.gate.outputs.should_publish == 'true'`.

### Pseudocode

```yaml
jobs:
  gate:
    runs-on: ubuntu-latest
    outputs:
      should_publish: ${{ steps.decide.outputs.should_publish }}
      reason: ${{ steps.decide.outputs.reason }}
    steps:
      - uses: actions/checkout@v6
        with:
          fetch-depth: 0
      - id: decide
        run: |
          TAG="${GITHUB_REF_NAME}"
          PREV=$(git describe --tags --abbrev=0 "${TAG}^" 2>/dev/null || echo "")
          MSG=$(git tag -l --format='%(contents)' "${TAG}")
          PUBLISH=false
          REASON="default: first tag, publishing"
          if [ -n "$PREV" ]; then
            CHANGED=$(git diff --name-only "$PREV" "$TAG" -- \
              'src/clarity/**' 'pyproject.toml')
            if [ -n "$CHANGED" ]; then
              PUBLISH=true
              REASON="code changed since $PREV"
            else
              PUBLISH=false
              REASON="no wheel-relevant changes since $PREV"
            fi
          fi
          if echo "$MSG" | grep -q '\[force-publish\]'; then
            PUBLISH=true
            REASON="forced by tag annotation"
          fi
          if echo "$MSG" | grep -q '\[skip-publish\]'; then
            PUBLISH=false
            REASON="skipped by tag annotation"
          fi
          echo "should_publish=$PUBLISH" >> "$GITHUB_OUTPUT"
          echo "reason=$REASON" >> "$GITHUB_OUTPUT"
          echo "::notice::Publish decision: $PUBLISH ($REASON)"

  check:
    uses: ./.github/workflows/build-and-docs.yml

  publish:
    needs: [gate, check]
    if: needs.gate.outputs.should_publish == 'true'
    # ... existing publish steps unchanged
```

`check` stays independent of `gate` so every tag still validates the
build + docs (catches breakage early, keeps the tag honest).

## GitHub release

Today the `publish` job does three things in sequence: PyPI upload,
generate release notes, create GitHub release.

Two options for the non-publish case:

### Option A (recommended): release only when publish runs

If we didn't publish a wheel, there is no artifact worth attaching
to a GitHub release. Do not create the GitHub release either. The tag
itself is the historical marker.

Pro: no "empty" GitHub releases, no drift between PyPI and GitHub.
Con: the release page stops tracking docs-only milestones.

### Option B: always create the release, skip only PyPI

Create the GitHub release unconditionally with `git-cliff` notes, but
only upload the wheel (and run `pypi-publish`) when `should_publish`.

Pro: the release page stays a full history.
Con: GitHub releases without artifacts look half-empty; consumers may
be confused.

Leaning **Option A**. If the project wants to keep docs-milestone
visibility, tag annotations already appear on the repo Tags page;
no need for a release record.

## Tag annotation conventions

To make overrides discoverable, document both markers in
`.prompt/instruction.md` and `docs/source/maintainer-workflow.rst`.

- `[force-publish]` -- rare; use to re-push a wheel after a failed
  PyPI upload when the underlying code hasn't changed.
- `[skip-publish]` -- rare; use to cut a milestone tag when code did
  change but the maintainer wants to delay the release.

Plain annotations without markers follow the path-diff default.

---

## Edge cases

1. **First tag ever** -- no previous tag to diff against. Behaviour:
   publish. (The `if [ -n "$PREV" ]` branch handles it.)
2. **Tag pointing at the same commit as the previous tag** -- the
   diff is empty. Behaviour: skip (no changes at all). Use
   `[force-publish]` if you actually want to republish.
3. **Light-weight tags** -- `git tag -l --format='%(contents)'` returns
   empty for light-weight (non-annotated) tags, so the override
   markers only work on annotated tags. That matches project policy:
   annotated + GPG-signed tags are required (`CLAUDE.md`).
4. **Re-tag after delete** -- if a tag is deleted and re-pushed, the
   workflow runs again. The gate will re-evaluate against the new
   previous tag. PyPI will reject a duplicate version upload, so the
   publish job fails loudly. Acceptable.
5. **Release notes (`git-cliff`)** -- the notes cover the commit range
   between the previous and current tag. When we skip publish we also
   skip notes; that is fine because there is no release page to fill.
6. **Matrix / parallel PRs** -- not a concern. `ci.yml` only runs on
   tag pushes.

## Alternatives considered

### A. Tag prefix split (`v*` = publish, `docs-v*` = skip)

Simpler CI, but it forces the maintainer to pick the tag shape up-
front, and `git-cliff` / other consumers expect a consistent pattern.
Rejected.

### B. Manual `workflow_dispatch` for publish

PyPI upload becomes a deliberate button click. Safer, but loses the
"tag and forget" flow. The user explicitly said tags should still be
pushable for milestones -- the automation should just be smarter, not
removed. Rejected.

### C. Pyproject-version-only check

Gate on `pyproject.toml`'s `version` field differing from the previous
PyPI wheel. Correct on intent but requires a PyPI API lookup in CI and
breaks when the maintainer forgets the version bump. The path-diff
approach is strictly more robust and needs no network.

### D. Semver-based heuristic (MAJOR / MINOR always publish, PATCH
   sometimes)

No reliable link between the numeric bump and the nature of the
change. Rejected.

## Verification plan

Before merging, exercise each branch on a throw-away tag in a scratch
fork to avoid polluting PyPI:

1. **Docs-only tag** -- change only `README.md`, tag `v1.4.0-001`,
   push. Expected: `gate` outputs `false`, `publish` skipped, no
   GitHub release created.
2. **Code tag** -- change `src/clarity/static/css/clarity.css`, tag
   `v1.4.1-000`. Expected: `gate` outputs `true`, PyPI upload runs,
   GitHub release created.
3. **Force tag** -- no code change, annotation includes
   `[force-publish]`. Expected: publish runs.
4. **Skip tag** -- code change, annotation includes `[skip-publish]`.
   Expected: publish skipped.
5. **First tag (no previous)** -- simulate by pointing `ci.yml` at a
   fresh repo; publish should run.

Once all five pass on the scratch fork, apply the change to main and
retire manual PyPI interventions.

## Rollout

- Phase 1: add the `gate` job behind `workflow_dispatch` for dry-run
  testing (outputs only, no publish dependency yet).
- Phase 2: wire `publish` to `gate.outputs.should_publish`.
- Phase 3: rename the publish job's `environment` to `pypi` (already
  named; no change) and confirm the Trusted Publisher still matches
  the file path (`.github/workflows/ci.yml`).
- Phase 4: update docs + release-policy notes.

## Open questions

1. **Include `LICENSE` in the trigger list?** Editing `LICENSE`
   arguably warrants a re-release because it changes the legal terms
   users get with the wheel. Lean yes -- add `LICENSE` to the trigger
   paths.
2. **Include `README.md` for PyPI long-description?** The wheel
   embeds `README.md` via the `readme` field. A README-only change
   does update what users see on the PyPI project page. Lean yes,
   promote it.
3. **Pre-release (`v0.*`) handling?** Already covered by the existing
   `PRERELEASE_FLAG`. No gate change needed.
4. **Should `CHANGELOG.md` tip-trigger?** No -- it is regenerated
   content, not a source for the wheel.

Final trigger-path list after resolving the open questions:

```
src/clarity/**
pyproject.toml
README.md
LICENSE
```

## Not in scope

- Auto-bumping the version in `pyproject.toml` (manual today; tracked
  elsewhere).
- Yanking bad releases from PyPI (manual via `twine`).
- Publishing pre-release channels (`a`, `b`, `rc`).
- Building multiple wheels (pure-Python, single-wheel project).
- Docs site deploy (separate workflow, not affected here).
