   * - High
     - deployer
     - ``src/clarity/static/js/chatbot.js``, ``docs/source/privacy.rst``
     - OpenRouter credentials are still persisted on the client and remain
       recoverable by any same-origin JavaScript. The current XOR obfuscation
       prevents casual inspection only; it does not materially protect against
       XSS or other script compromise on the same origin.
     - Keep this risk explicitly accepted only for low-value keys, document the
       threat model wherever chatbot setup is explained, and avoid encouraging
       management-key persistence as the default path.
     - Accepted risk
   * - Low
     - deployer
     - ``src/clarity/theme.conf``, ``src/clarity/static/js/consent.js``,
       ``docs/source/privacy.rst``
     - The default font stack still triggers Google Fonts requests after
       consent, which may be incompatible with strict CSP or privacy-sensitive
       deployments.
     - Keep ``font_stack = "system"`` as the documented no-remote option and
       surface the tradeoff earlier in install/config docs.
     - Acknowledged

Refactoring Proposal
--------------------

.. list-table::
   :header-rows: 1
   :widths: 8 12 20 32 18 10

   * - Priority
     - Audience
     - Code target
     - Current pain
     - Proposed refactor
     - Status
   * - Medium
     - maintainer
     - ``src/clarity/layout.html``, ``src/clarity/static/js/chatbot.js``
     - ``chatbot_enabled`` hides the markup, but chatbot CSS and JavaScript are
       still shipped on every page. The runtime no-ops safely, but the payload
       is still paid even when the feature is disabled.
     - Gate chatbot asset includes on ``theme_chatbot_enabled`` or move the
       feature into a conditional bundle.
     - Resolved
   * - Medium
     - maintainer
     - ``src/clarity/static/js/chatbot.js``,
       ``src/clarity/static/js/chatbot-storage.js``,
       ``src/clarity/static/js/chatbot-api.js``,
       ``src/clarity/static/js/chatbot-render.js``,
       ``src/clarity/static/js/chatbot-commands.js``
     - The current file shares mutable state across persistence, network,
       rendering, and command execution concerns, which makes future changes
       more coupled than they need to be.
     - Extract storage/API/render/commands layers into narrower internal units
       and keep the boot code as the only DOM orchestration layer.
     - Resolved

Removed After Verification
--------------------------

- ``navigation_with_keys`` is wired through ``layout.html`` and consumed by
  ``clarity.js``.
- ``sidebar_hide_name`` is implemented in ``layout.html``.
- Ordered-list rendering in the chatbot markdown path now tracks list type and
  closes the correct tag.
- The management key is no longer required in the chatbot save flow.
- Chatbot CSS and JS are now gated on ``theme_chatbot_enabled`` in
  ``layout.html``, so the payload is no longer shipped when the feature is
  disabled.
- ``chatbot.js`` is now split into ``chatbot-storage.js``, ``chatbot-api.js``,
  ``chatbot-render.js``, ``chatbot-commands.js``, and a slim boot file that
  only does DOM orchestration. Modules attach to
  ``window.Clarity.Chatbot.{storage, api, render, commands}``.
